# FitFocus: מדריך עברי מפורט להסבר הקוד בפני המרצה

המסמך הזה נועד להכין אותך לשאלות בעל-פה על פרויקט FitFocus.
המטרה שלו היא להסביר מה כל אזור חשוב בקוד עושה, למה הוא קיים, ומה הלוגיקה שמאחוריו.

המסמך מבוסס על הקוד הקיים כרגע בריפו, ולא רק על הרעיון המקורי של הפרויקט.
הוא מתמקד בקוד המקור ובארכיטקטורה, ולא בקבצים שנוצרים אוטומטית כמו `package-lock.json`, אייקונים או קבצי assets.

## 1. סקירה כללית של המערכת

FitFocus היא מערכת לקוח-שרת למעקב בריאות והרגלי חיים, עם שני חלקים מרכזיים:

1. `FitFocus.Mobile`
   זהו צד הלקוח במובייל, שנבנה ב-React Native + Expo.
   הוא מציג את הממשק, שומר את הסשן של המשתמש מקומית, מבצע קריאות ל-API, מנהל תזכורות מקומיות ומציג מידע שבועי.

2. `FitFocus.Api`
   זהו צד השרת, שנבנה ב-ASP.NET Core Web API.
   הוא מטפל באימות משתמשים, הרשאות, ולידציה, גישה למסד הנתונים, אבטחה, אגרגציה של מידע ושליחת התראות Push.

הפרויקט בנוי עם הפרדת אחריות ברורה:

1. אפליקציית המובייל לא פונה ישירות ל-SQL Server.
2. השרת לא מייצר UI.
3. כל הגישה למסד הנתונים נעשית דרך שכבת Repositories.
4. מצב האימות עובר באמצעות JWT.

הפרדה כזו חשובה כי היא הופכת את המערכת ליותר מסודרת, יותר קלה להסבר, ויותר קלה לתחזוקה.

## 2. זרימת עבודה מקצה לקצה

לפני שנכנסים קובץ-קובץ, זה המודל המחשבתי הכי חשוב לזכור.

### 2.1 זרימת התחברות

1. המשתמש מקליד מייל וסיסמה במסך ההתחברות במובייל.
2. `FitFocus.Mobile/src/screens/AuthScreen.tsx` קורא ל-`api.login(...)`.
3. `FitFocus.Mobile/src/api.ts` שולח `POST /api/auth/login`.
4. `FitFocus.Api/Controllers/AuthController.cs` מחפש את המשתמש לפי המייל.
5. `PasswordService` בודק שהסיסמה תואמת ל-hash השמור.
6. אם ה-hash הישן הוא בפורמט ישן, השרת משדרג אותו אוטומטית ל-PBKDF2.
7. `TokenService` מייצר JWT.
8. המובייל שומר את הסשן ב-`AsyncStorage`.
9. כל קריאת API עתידית מצרפת אוטומטית `Authorization: Bearer <token>` דרך Axios interceptor.

### 2.2 זרימת יומן יומי

1. המובייל מחשב את התאריך של היום בפורמט `YYYY-MM-DD`.
2. מסך היומן היומי טוען רשומה קיימת של היום אם יש כזאת.
3. בלחיצה על שמירה, האפליקציה שולחת `POST /api/dailylogs`.
4. השרת בודק האם כבר קיימת רשומה עבור אותו משתמש ואותו תאריך.
5. אם קיימת רשומה, ה-controller מעדכן אותה.
6. אם לא קיימת רשומה, ה-controller יוצר אחת חדשה.

כלומר, הצד של המובייל לא צריך לדעת האם זו יצירה או עדכון.
מבחינת המשתמש זו פשוט פעולה של "שמור את היומן של היום".

### 2.3 זרימת דשבורד

1. הדשבורד מחשב את השבוע המקומי הנוכחי מיום ראשון עד שבת.
2. הוא שולח בקשות ל-`/api/dailylogs/range?from=...&to=...`, ל-`/api/profile`, ולנתוני ארוחות שבועיים דרך `/api/meals/range?from=...&to=...`.
3. השרת מחזיר את הלוגים, פרטי הפרופיל והארוחות של המשתמש עבור אותו טווח.
4. המובייל הופך את המידע הזה לנתונים עבור גרפים של קלוריות, מצב רוח, שינה, לחץ ומים, וגם לסיכומים שבועיים בטקסט.

הנקודה החשובה כאן היא שהתאריכים מטופלים כתאריכים מקומיים, ולא כ-UTC timestamp.
הפתרון הזה מונע מצב שבו מידע של יום ראשון מוצג מתחת ליום שבת.

### 2.4 זרימת ארוחות

1. המשתמש ממלא פרטי ארוחה, ויכול גם לצרף תמונה.
2. האפליקציה מנסה, אם צריך, להביא את מזהה היומן היומי של היום.
3. היא שולחת `POST /api/meals`.
4. השרת שומר את הארוחה עם ה-`UserId` שמופק מה-JWT.
5. מסך ההיסטוריה של הארוחות נטען מחדש, וניתן לסנן לפי תאריך וסוג ארוחה.

### 2.5 זרימת תזכורות

1. המשתמש יוצר תזכורת לתרופה דרך האפליקציה.
2. התזכורת נשמרת בשרת לצורך persistence.
3. בנוסף, האפליקציה מתזמנת התראה מקומית במכשיר דרך Expo Notifications.

חשוב להבין: שמירת התזכורת ותזמון ההתראה הם שני דברים שונים.
שמירת התזכורת היא אחריות של השרת.
תזמון ההתראה עצמה הוא אחריות של המכשיר.

## 3. מדריך לצד המובייל

## 3.1 `FitFocus.Mobile/App.tsx`

זהו קובץ ה-root של אפליקציית המובייל.

התפקידים המרכזיים שלו הם:

1. אתחול סשן המשתמש.
2. בניית הניווט באפליקציה.
3. חיבור התנהגות גלובלית של אימות.
4. טיפול ב-push readiness.

### הלוגיקה המרכזית

1. `SESSION_KEY` מגדיר איפה הסשן נשמר בתוך `AsyncStorage`.
2. `setAuthenticated` שומר את ה-JWT בזיכרון דרך `api.setToken(...)` וגם שומר את הסשן מקומית.
3. `logout` מנקה גם את הטוקן מהזיכרון וגם את המידע השמור.
4. `useEffect` אחד רץ בטעינה הראשונית:
   הוא מנסה לשחזר סשן קודם, בודק אותו דרך `api.getProfile()`, ואם הטוקן כבר לא תקף הוא מבצע logout.
5. `useEffect` נוסף שומר על סנכרון בין מצב ההתחברות לבין מצב ההתראות.
6. `useEffect` שלישי מבטל התראות מקומיות כאשר המשתמש מכבה את ההתראות.

### למה המבנה הזה נבחר

ה-root component צריך להיות בעל הבית של נושא האימות, כי האימות משפיע על כל האפליקציה.
אם כל מסך היה מנהל את מצב ההתחברות לבד, הקוד היה הופך מהר מאוד ללא עקבי.

### לוגיקת ניווט

יש כאן שתי רמות:

1. Stack navigator:
   הוא מחליף בין `Auth` לבין `Main`, והוא גם מכיל את מסך `Admin` הייעודי.

2. Bottom tab navigator:
   הוא מציג את החלקים המרכזיים של האפליקציה לאחר התחברות: `Dashboard`, `Daily Log`, `Meals`, `Reminders`, ו-`Profile`.

המסך `Admin` כבר לא מופיע כלשונית קבועה בתחתית.
הוא נפתח מתוך `Profile` ורק כאשר `session.role === "Admin"`.
זו החלטת UI.
האבטחה האמיתית עדיין נאכפת בשרת דרך role-based authorization.

## 3.2 `FitFocus.Mobile/src/api.ts`

זהו קובץ שכבת התקשורת בין המובייל לבין השרת.
זה אחד הקבצים החשובים ביותר בכל הפרויקט.

### מה הקובץ עושה

1. קובע את כתובת ה-API המתאימה.
2. מגדיר את ה-types של התגובות מהשרת.
3. יוצר Axios instance אחד מרכזי.
4. מצרף אוטומטית את ה-JWT לכל בקשה.
5. מטפל ב-`401 Unauthorized` באופן גלובלי.
6. חושף את כל פעולות ה-API כפונקציות פשוטות.

### לוגיקת בחירת Base URL

האפליקציה בוחרת את כתובת ה-API לפי הסדר הבא:

1. override מהסביבה או מ-`app.config.js`
2. Expo host URI
3. fallback לאמולטור אנדרואיד `10.0.2.2`
4. fallback ל-`localhost`

זה חשוב כי הפרויקט צריך לעבוד במצבים שונים:

1. טלפון אמיתי
2. אמולטור
3. מחשב מקומי
4. סביבת הדגמה שונה

### לוגיקת Interceptors

ה-request interceptor מוסיף טוקן אם קיים כזה.

ה-response interceptor מאזין ל-`401`.
אם מתקבל `401`, הוא מפעיל callback גלובלי שנרשם מתוך `App.tsx`, שמבצע logout מרוכז.

זה הרבה יותר נקי מאשר לנסות לטפל ב-expired token בכל מסך בנפרד.

### קבוצות ה-API החשובות

1. Authentication:
   `register`, `login`

2. Profile:
   `getProfile`, `updateProfile`

3. Daily logs:
   `upsertDailyLog`, `getDailyLog`, `getDailyLogRange`

4. Meals:
   `createMeal`, `getMeals`, `getMealsRange`, `getMealsByLog`, `deleteMeal`

5. Reminders:
   `getReminders`, `createReminder`, `deleteReminder`

6. Notifications:
   `registerDeviceToken`, `unregisterDeviceToken`, `sendTestPush`

7. Admin:
   `getAdminUsers`

הדשבורד במובייל מורכב כיום מ-range endpoints ומנתוני פרופיל, ולא ממתודה אחת בשם `getDashboard`.

## 3.3 `FitFocus.Mobile/src/types/index.ts`

הקובץ הזה מגדיר את הטיפוס `Session`.

הוא קיים כדי להפוך את מצב האימות למפורש ול-safe מבחינת טיפוסים.
בלי קובץ כזה, הסשן היה אובייקט JSON "חופשי" והיה יותר קל לטעות בשימוש בו.

## 3.4 קבצי Utilities במובייל

### `FitFocus.Mobile/src/utils/dateUtils.ts`

הקובץ הזה קיים כי תאריך מקומי ו-UTC timestamp הם לא אותו דבר.

הוא מספק:

1. `formatLocalDate`
2. `getTodayLocalDate`
3. `startOfLocalWeek`
4. `addDays`

הרעיון המרכזי הוא שהשרת שומר תאריכים לוגיים כמו `2026-04-12`, ולא רגעים מוחלטים בציר הזמן.
אם האפליקציה הייתה משתמשת ב-`toISOString().slice(0, 10)`, תאריך חצות מקומי בישראל היה עלול לזוז אחורה ליום הקודם ב-UTC.

הקובץ הזה מרכז את לוגיקת התאריכים ומונע באגים מהסוג הזה.

### `FitFocus.Mobile/src/utils/errorUtils.ts`

הקובץ הזה מאחד את הדרך שבה האפליקציה מציגה שגיאות.

המטרה שלו היא למנוע מצב שבו כל מסך כותב parsing משלו לשגיאות.

הוא יודע להתמודד עם:

1. כשלי רשת
2. תגובות שרת שהן מחרוזת
3. validation errors
4. תגובות בסגנון `ProblemDetails`
5. fallback message

זה חשוב במיוחד בדמו, כי זה הופך שגיאות מעורפלות להודעות ברורות.

### `FitFocus.Mobile/src/utils/notificationUtils.ts`

הקובץ הזה מרכז את כל לוגיקת ההתראות.

הפונקציות החשובות:

1. `setupNotifications`
   מגדירה איך Expo יציג התראות כשהאפליקציה פתוחה.

2. `scheduleLocalReminder`
   מתזמנת התראה מקומית יומית על המכשיר.

3. `registerNotifications`
   בודקת הרשאות, מוודאת שמדובר במכשיר אמיתי, מייצרת Expo push token ומחזירה אותו.

4. `ensurePushReady`
   רושמת את ה-device token מול השרת, או מוחקת אותו אם המשתמש כיבה התראות.

### למה הלוגיקה הזו חשובה

התראות הן לא רק פיצ'ר של UI.
הן מערבות:

1. הרשאות
2. יכולות חומרה
3. תזמון מקומי על המכשיר
4. שמירת token בשרת
5. שליחת push מרחוק

ריכוז כל זה בקובץ אחד מונע כפילויות והופך את הזרימה לברורה להסבר.

## 3.5 קומפוננטות Reusable במובייל

הקומפוננטות האלו קיימות כדי לשמור את המסכים יותר קטנים, מסודרים ועקביים.

### `ActionButton.tsx`

זוהי אבסטרקציה של כפתור על בסיס `Pressable`.
היא תומכת ב-`primary`, `secondary` ו-`danger`.

למה היא קיימת:

1. אחידות ויזואלית
2. פחות שכפול של styles
3. קל יותר לשנות את ה-UI בעתיד

### `InputField.tsx`

הקומפוננטה הזו עוטפת `TextInput` עם label ועם style משותף.

לוגיקה חשובה:
היא מכריחה יישור שמאלה עבור שדות סיסמה ואימייל.

זה חשוב במיוחד בסביבת RTL בעברית, שבה אימיילים וסיסמאות עדיין צריכים להתנהג כתוכן LTR.

### `SliderInput.tsx`

הקומפוננטה הזו עוטפת slider עם label ועם preview של הערך בזמן אמת.

היא משמשת למדדים כמו:

1. מצב רוח
2. לחץ
3. שינה
4. מים
5. קלוריות
6. גובה
7. משקל

### `SegmentedPicker.tsx`

זהו segmented control שמבוסס על `Pressable`.
הוא נותן בחירה נוחה ונקייה עבור סט ערכים סגור, כמו סוג ארוחה או מגדר.

### `FilterChip.tsx`

זו קומפוננטה קטנה עבור פילטרים בצורת chips במסך ההיסטוריה של הארוחות.

### `ChartBox.tsx`

הקומפוננטה הזו מציגה כרטיס גרף.
האחריות שלה היא:

1. להציג את הערך של היום
2. להראות סימון של `Today`
3. לצייר את הגרף עצמו
4. להציג את נקודת הנתון שנבחרה כאשר המשתמש לוחץ על הגרף

הרעיון המרכזי הוא שכל לוגיקת התצוגה של הגרף תהיה מרוכזת כאן, ולא בתוך הדשבורד עצמו.

## 3.6 מסכי המובייל

### `AuthScreen.tsx`

המסך הזה מטפל גם בהרשמה וגם בהתחברות.

### מה הוא עושה

1. מחזיק state של מצב התחברות/הרשמה
2. מנהל את שדות הטופס
3. קורא ל-`api.register(...)` או `api.login(...)`
4. שולח את אובייקט הסשן ל-`App.tsx`

### למה הלוגיקה הזו חשובה

זה לא רק מסך שמקבל שם משתמש וסיסמה.
הוא גם עוזר לאמינות של הדמו, כי הוא מראה לפני השימוש אם האפליקציה באמת מצליחה לתקשר עם השרת.

### `DailyLogScreen.tsx`

זהו המסך של המדדים הבריאותיים היומיים.

### מה הוא עושה

1. מחשב את תאריך היום המקומי
2. טוען את הרשומה הקיימת של היום אם יש כזו
3. ממלא את השדות מראש אם כבר קיימת רשומה
4. שולח save אחד בסגנון upsert

### הלוגיקה מאחורי העיצוב

המשתמש אמור לחשוב במונחים של "היומן של היום", ולא במונחים של "רשומה מספר 43".
לכן המסך מבוסס תאריך ולא מבוסס מזהה.

### `DashboardScreen.tsx`

זהו המסך עם הכי הרבה לוגיקה בצד המובייל.

### מה הוא עושה

1. מרענן מידע שבועי, ארוחות שבועיות ונתוני פרופיל כאשר המסך מקבל focus
2. מחשב את השבוע הנוכחי מיום ראשון עד שבת לפי תאריך מקומי
3. בונה נתונים לגרפים של קלוריות, מצב רוח, שינה, לחץ ושתיית מים
4. משווה בין צריכת קלוריות לבין קו של estimated daily burn כאשר יש מספיק נתוני פרופיל
5. מציג את הגרפים בעמודה אחת ממורכזת כדי שיהיה נוח יותר לעין
6. מסמן את היום הנוכחי גם בגרפים וגם בכרטיסי הטקסט השבועיים
7. מציג סיכומים שבועיים של `Symptoms` ושל `Notes`

### למה הלוגיקה מסודרת כך

הדשבורד נבנה סביב השוואה מבוססת זמן.
לכן הוא צריך:

1. גבולות שבוע קבועים
2. יצירת labels עקבית
3. התאמה נכונה בין תאריכים לבין נתונים

## 4. מדריך לצד השרת

## 4.1 `FitFocus.Api/Program.cs`

זהו קובץ ה-composition root של השרת.
כאן מחברים את כל המערכת.

### תחומי האחריות המרכזיים

1. רישום services ו-repositories ל-DI
2. קונפיגורציה של JWT authentication
3. קונפיגורציה של authorization
4. קונפיגורציה של CORS
5. קונפיגורציה של exception handling גלובלי
6. חשיפת `/health`
7. הפעלת bootstrap של מסד הנתונים ברקע

### לוגיקת Dependency Injection

Infrastructure ו-services נרשמים כ-singletons כאשר זה מתאים.
Repositories נרשמים כ-scoped, כי הם מייצגים התנהגות גישה לנתונים ברמת הבקשה.

### לוגיקת JWT

`JwtOptions` נטענים מתוך configuration.
מפתח החתימה הופך ל-`SymmetricSecurityKey`.
JWT bearer authentication מוגדר עם:

1. בדיקת issuer
2. בדיקת audience
3. בדיקת signing key
4. בדיקת expiry
5. `ClockSkew = 0`

`ClockSkew = 0` הופך את ההתנהגות של תוקף הטוקנים ליותר קשיחה ויותר דטרמיניסטית.

### לוגיקת CORS

Allowed origins יכולים להגיע מ:

1. environment variable בשם `FITFOCUS_ALLOWED_ORIGINS`
2. section בקונפיגורציה בשם `Cors:AllowedOrigins`
3. ערכי development ברירת מחדל

כך האפליקציה נהיית יותר portable בלי לגעת בקבצי config במעקב.

### לוגיקת טיפול בשגיאות

האפליקציה משתמשת ב-`UseExceptionHandler`.
במקום לקרוס או להחזיר שגיאות לא עקביות, היא ממפה כשלים ידועים ל-`ProblemDetails`.

לדוגמה:

1. `SqlException` נהפך ל-"Database unavailable"
2. `TimeoutException` נהפך ל-"Request timed out"
3. כל exception אחר נהפך לשגיאת שרת כללית

זה נחשב ארכיטקטורה טובה כי controllers נשארים ממוקדים בלוגיקה העסקית, וה-pipeline מטפל ב-cross-cutting failures ברמה מרכזית.

### לוגיקת Startup Resilience

השרת כבר לא נכשל לגמרי אם מסד הנתונים לא זמין בזמן עלייה.
במקום זה:

1. ה-web server עולה
2. ה-bootstrap של הדאטהבייס רץ ברקע
3. הסטטוס נשמר ב-`StartupStatus`
4. `/health` מחזיר האם הדאטהבייס מוכן

זו נקודת שיפור ארכיטקטונית משמעותית, כי האפליקציה יכולה להיכנס למצב degraded במקום לקרוס.

### `SqlConnectionFactory.cs`

2. אם הקובץ ריק, היא רק כותבת warning ומדלגת

היא קיימת כדי ש-`/health` יוכל להחזיר סטטוס ברור.

## 4.3 Controllers בשרת

### `AuthController.cs`

ה-controller הזה מטפל בהרשמה ובהתחברות.

#### Register

1. בודק האם המייל כבר קיים
2. מנרמל את המייל ל-lowercase
3. מבצע hash לסיסמה
4. יוצר משתמש חדש
5. טוען מחדש את המשתמש שנוצר
6. מייצר JWT
7. מחזיר `AuthResponse`

#### Login

1. מחפש את המשתמש לפי מייל מנורמל
2. בודק את הסיסמה
3. אם צריך, משדרג את ה-hash הישן לפורמט החדש
4. מייצר JWT
5. מחזיר את ה-auth payload

ה-automatic rehash הוא תשובה מצוינת אם המרצה ישאל איך שיפרתם אבטחה בלי לשבור משתמשים קיימים.

### `ProfileController.cs`

ה-controller הזה מחזיר ומעדכן את הפרופיל של המשתמש המחובר.

הוא אף פעם לא מקבל `userId` מהלקוח.
במקום זה, הוא מפיק את `userId` מתוך ה-JWT.

זו החלטת אבטחה חשובה כי היא מונעת מלקוח לנסות לעדכן פרופיל של משתמש אחר על ידי ניחוש מזהה.

### `DailyLogsController.cs`

ה-controller הזה מטפל ב-CRUD של היומן היומי.

הלוגיקה המרכזית נמצאת ב-`POST`:

1. לחפש האם קיימת רשומה עבור אותו תאריך
2. אם כן, לעדכן אותה
3. אם לא, ליצור אחת חדשה

כך המובייל נשאר פשוט, והכלל "רשומה אחת למשתמש לכל יום" נאכף בצורה עקבית.

### `DashboardController.cs`

ה-controller הזה בונה סיכום עבור מספר ימים אחרונים.

הוא:

1. מגביל את מספר הימים המבוקש
2. מחשב טווח תאריכים
3. טוען logs
4. סופר ארוחות
5. מעביר את הכל ל-`RiskScoreService`

זו דוגמה טובה ל-controller נכון:
ה-controller מרכז תיאום של הנתונים, אבל הלוגיקה העסקית עצמה נמצאת ב-service.

### `MealsController.cs`

ה-controller הזה מספק:

1. שליפת ארוחות עם פילטרים
2. יצירת ארוחה
3. שליפה לפי daily log
4. מחיקה

גם כאן, ה-`UserId` מגיע תמיד מה-JWT ולא מתוך payload שהלקוח שלח.

### `RemindersController.cs`

ה-controller הזה מנהל תזכורות persistent.

זהו CRUD קלאסי עבור `MedicationReminder`.

### `NotificationsController.cs`

ה-controller הזה מנהל device tokens ו-test push notifications.

#### Register / Unregister

האפליקציה יכולה להוסיף או להסיר Expo push token עבור המשתמש המחובר.

#### Send Test

השרת:

1. שולף את כל ה-device tokens של המשתמש
2. בונה payload עבור Expo
3. שולח אותו דרך `HttpClient`
4. סופר כמה נשלחו בהצלחה
5. אוסף פירוט של כישלונות

ה-endpoint הזה מאוד שימושי בדמו כי הוא מוכיח שהשרת יודע ליזום outbound push.

### `AdminController.cs`

ה-controller הזה חושף רשימת משתמשים למנהל בלבד.

הנקודה החשובה היא לא רק המידע שהוא מחזיר.
הנקודה החשובה היא ההגבלה:

`[Authorize(Roles = "Admin")]`

זו שכבת האכיפה האמיתית של role-based access control.

## 4.4 Services בשרת

### `PasswordService.cs`

זהו אחד מקבצי האבטחה החשובים ביותר בפרויקט.

### הלוגיקה הנוכחית

1. סיסמאות חדשות נשמרות באמצעות PBKDF2 עם SHA-256
2. פורמט ה-hash כולל:
   prefix, iteration count, salt ו-hash
3. `VerifyPassword` תומך גם בפורמט החדש וגם בפורמט legacy
4. `NeedsRehash` מזהה מתי צריך לשדרג hash קיים

### למה זה חזק

המערכת משפרת את אבטחת הסיסמאות בלי לחייב את כל המשתמשים הקיימים לאפס סיסמה.

### `TokenService.cs`

הקובץ הזה מייצר JWT token.

הוא מכניס ל-token:

1. מזהה משתמש
2. מייל
3. שם מלא
4. role

ה-claim של role הוא מה שמאפשר ל-endpoints של אדמין לעבוד.

### `RiskScoreService.cs`

ה-service הזה מתרגם התנהגות בריאותית אחרונה לסיכום risk.

### הלוגיקה שבה הוא משתמש

רמת הסיכון עולה כאשר:

1. ממוצע שעות השינה נמוך מדי
2. ממוצע מצב הרוח נמוך
3. ממוצע הלחץ גבוה
4. שתיית המים נמוכה
5. דווחו סימפטומים בתדירות גבוהה

לאחר מכן ה-service:

1. מגביל את הציון לטווח `0..100`
2. ממפה אותו לרמה:
   `Low`, `Medium`, `High`, `Critical`
3. מחזיר גם הסברים ב-`RiskSignals`

זו דוגמה מצוינת ל-service layer, כי זו לוגיקה עסקית טהורה שלא תלויה ב-HTTP או ב-SQL.

## 4.6 Interfaces ו-Contracts בשרת

4. `FitFocus_MedicationRemindersApp`
5. `FitFocus_DeviceTokensApp`

### החלטות סכימה חשובות

1. למשתמש יש מייל ייחודי
2. ליומן יומי יש unique constraint על `(UserId, LogDate)`
3. ארוחות יכולות להצביע אופציונלית ל-daily log
4. device tokens ייחודיים לפי `(UserId, ExpoPushToken)`

ה-unique constraint על היומן היומי חשוב במיוחד.
הוא אוכף ברמת מסד הנתונים את אותו כלל עסקי שה-controller כבר מיישם.

## 5. קבצי עזר חשובים בשרת

### `ClaimsPrincipalExtensions.cs`

זהו helper שמחלץ את מזהה המשתמש מתוך ה-claims.

למה זה חשוב:

ה-controllers לא צריכים לשכפל שוב ושוב את לוגיקת קריאת ה-claims.
כך הקוד נשאר נקי יותר ופחות מועד לטעויות.

### `FitFocus.Api.csproj`

הקובץ הזה מגדיר את פרויקט השרת עצמו ואת תלויותיו.

### `FitFocus.Api.http`

זהו קובץ עזר עבור בדיקות ידניות של ה-API בזמן פיתוח.

## 6. החלטות ארכיטקטוניות ולמה הן נכונות

## 6.1 למה Repository Pattern?

כי controllers לא אמורים לדעת את פרטי ה-SQL.

התפקיד של controller:
לקבל בקשה, להפעיל אימות והרשאות, ולתאם פעולות.

התפקיד של repository:
לדבר עם מסד הנתונים.

ההפרדה הזו הופכת את הקוד ליותר מובן ויותר קל להסבר בשכבות.

## 6.2 למה JWT?

כי אפליקציות מובייל צריכות authentication stateless שעובד על פני הרבה בקשות בלי לשמור session server-side.

JWT מאפשר:

1. שימוש פשוט ב-Authorization headers
2. role claims
3. סקיילינג פשוט יותר עקרונית

## 6.3 למה לשמור session ב-AsyncStorage?

כי באפליקציית מובייל חייבים persistence בין פתיחה לסגירה של האפליקציה.
בלי זה המשתמש היה צריך להתחבר כל פעם מחדש.

## 6.4 למה ADO.NET ולא Entity Framework?

כי הפרויקט רוצה שליטה ישירה ומפורשת על ה-SQL.

זו תשובה טובה אם המרצה ישאל למה לא השתמשתם ב-ORM.

## 6.5 למה גם Local Reminders וגם Push Notifications?

כי הם פותרים בעיות שונות:

1. local reminders פועלים על המכשיר בזמן קבוע מראש
2. push notifications מאפשרים לשרת ליזום תקשורת

שימוש בשניהם מדגים כיסוי יותר רחב של יכולות מובייל.

## 6.6 למה ה-API עולה במצב Degraded?

כי כישלון מלא בזמן startup הוא סיפור תפעולי חלש.

עם degraded mode:

1. ה-API עדיין יכול לחשוף `/health`
2. ה-frontend יכול להבין מה שבור
3. אפשר לאבחן את השרת גם אם SQL לא זמין

## 7. נקודות חזקות לשאלות בעל-פה

אם המרצה ישאל "למה עשיתם את זה כך?", אלו תשובות טובות.

### Authentication

האפליקציה משתמשת ב-JWT כי הלקוח הוא מובייל וצריך authentication stateless.
הסיסמאות נשמרות עם PBKDF2, ו-hashes ישנים משודרגים אוטומטית בהתחברות מוצלחת.

### Dashboard

הדשבורד מבוסס שבוע ועובד עם local calendar dates ולא עם UTC strings, כדי למנוע זליגות של יום קדימה או אחורה.
הוא מחשב טווח יציב מיום ראשון עד שבת ובונה structures שמתאימים לגרפים.

### עקביות נתונים

ליומן היומי יש ייחוד לפי משתמש ולפי תאריך גם בלוגיקת ה-controller וגם ב-unique constraint ב-SQL.
זו דוגמה טובה ל-defense in depth.

### Notifications

האפליקציה לא מניחה שהתראות יעבדו.
היא בודקת הרשאות, דרישת physical device, יצירת Expo token ורישום token בשרת.

### אמינות

השרת כבר לא קורס לגמרי אם SQL לא זמין בזמן startup.
זה הופך את המערכת ליותר אמינה וליותר מתאימה להדגמה.

## 8. שאלות שהמרצה כנראה ישאל ותשובות חזקות

### "מה התפקיד של `App.tsx`?"

זהו קובץ ה-composition root של המובייל.
הוא משחזר סשן, מחבר לוגיקת auth גלובלית, מגדיר notifications ובונה את היררכיית הניווט.

### "למה יש `api.ts` ולא קריאות Axios ישירות מכל מסך?"

כי נושאים כמו base URL, token injection ו-unauthorized handling הם cross-cutting concerns.
ריכוז שלהם במקום אחד מונע כפילות וחוסר עקביות.

### "איפה הלוגיקה העסקית?"

רוב הלוגיקה העסקית נמצאת ב-backend services וב-controllers.
למשל, חישוב risk score נמצא ב-`RiskScoreService`, והלוגיקה של create-or-update ליומן יומי נמצאת ב-`DailyLogsController`.

### "איך אתם מונעים ממשתמש אחד לראות מידע של משתמש אחר?"

השרת מפיק את ה-`userId` מתוך ה-JWT באמצעות `ClaimsPrincipalExtensions.GetUserId()`.
ה-repositories תמיד מקבלים את ה-`userId` הזה, ולא `userId` שנשלח מהלקוח.

### "למה התראות מטופלות ביותר ממקום אחד?"

כי חלק אחד הוא persistence וחלק שני הוא device behavior.
השרת שומר tokens ויודע לשלוח push.
האפליקציה מטפלת בהרשאות ובתזמון מקומי.

### "למה יצרתם קומפוננטות reusable כמו `InputField` ו-`ActionButton`?"

כדי להקטין את המסכים, לצמצם שכפול styles ולשמור על UI אחיד.

### "איך האפליקציה יודעת לאיזו כתובת API לפנות?"

`api.ts`, `app.config.js` ו-`start-expo.cjs` עובדים יחד.
המערכת יכולה להשתמש ב-environment override, ב-Expo host detection, ב-emulator fallback או ב-localhost fallback, לפי הסביבה.

### "מה השיפורים הארכיטקטוניים הבולטים בגרסה הנוכחית?"

1. עלייה עמידה יותר של השרת
2. שיפור hashing של סיסמאות
3. קונפיגורציית API מרכזית במובייל
4. טיפול שגיאות עקבי יותר
5. תיקון לוגיקת התאריכים המקומיים בדשבורד

## 9. מה כדאי להדגיש בזמן דמו

אם המרצה יבקש ממך להסביר את הארכיטקטורה תוך כדי הדגמה, כדאי לדבר לפי הסדר הזה:

1. ההתחברות מבוססת JWT והסשן נשמר ב-local storage
2. האפליקציה מדברת רק עם השרת, לא ישירות עם SQL
3. היומן היומי עובד לפי תאריך ועם create-or-update
4. מסך הארוחות תומך בתמונות ובהיסטוריה עם פילטרים
5. התזכורות משלבות שמירה בשרת עם תזמון מקומי
6. הדשבורד בונה תצוגה שבועית מנתונים אמיתיים שנשמרו
7. השרת כולל auth, repositories, services, health checks ו-push token support

## 10. סיכום

קוד הבסיס של FitFocus בנוי סביב הפרדת אחריות.

המובייל אחראי על תצוגה, מצב מקומי, יכולות מכשיר וצריכת API.
השרת אחראי על אבטחה, לוגיקה עסקית, גישה למסד הנתונים ותיאום מערכתי.

הנקודות החשובות ביותר להבין לשאלות בעל-פה הן:

1. ניווט ואתחול סשן ב-`App.tsx`
2. שכבת תקשורת מרכזית ב-`api.ts`
3. לוגיקת תאריכים מבוססת יום בדיילי לוג ובדשבורד
4. גישה ל-SQL דרך repositories
5. JWT authentication והרשאות לפי claims
6. PBKDF2 עם legacy migration עבור סיסמאות
7. startup resilience עם health reporting

אם תדע להסביר היטב את הנקודות האלו, תוכל לענות בצורה חזקה על רוב השאלות הארכיטקטוניות והטכניות על הפרויקט.
